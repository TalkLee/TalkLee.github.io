浏览器工作原理

###浏览器工作原理

本文主要是学习
浏览器的工作原理：新式网络浏览器幕后揭秘
时做的笔记，期间还参杂自己的一些理解。

####浏览器功能:

向服务器发出请求，获取以及发送一些网络资源，主要是html文档，当然还包括css，js，pdf,多媒体以及其他类型文件。资源位置由URI指定。

用户界面(基本元素):

  • URL地址栏
  • 前进后退
  • 书签功能
  • 刷新和停止刷新
  • 主页按钮

浏览器结构(非本文重点,直接引用)

用户界面

  • 包括地址栏、前进/后退按钮、书签菜单等。除了浏览器主窗口显示的您请求的页面外，其他显示的各个部分都属于用户界面。

浏览器引擎

  • 在用户界面和呈现引擎之间传送指令。

呈现引擎

  • 负责显示请求的内容。如果请求的内容是 HTML，它就负责解析 HTML 和 CSS 内容，并将解析后的内容显示在屏幕上。

网络

  • 用于网络调用，比如 HTTP 请求。其接口与平台无关，并为所有平台提供底层实现。

用户界面后端

  • 用于绘制基本的窗口小部件，比如组合框和窗口。其公开了与平台无关的通用接口，而在底层使用操作系统的用户界面方法。

JavaScript 解释器

  • 用于解析和执行 JavaScript 代码。

数据存储

  • 这是持久层。浏览器需要在硬盘上保存各种数据，例如 Cookie。新的 HTML 规范 (HTML5) 定义了“网络数据库”，这是一个完整（但是轻便）的浏览器内数据库。

####渲染引擎

用于显示浏览器请求的内容，默认可以显示html，css以及图片，通过插件等方式可以支持其他格式文件显示。

主要有以下几种渲染引擎(内核)：

  • Trident(IE使用，Edge新内核为EdgeHTML,具体内容如何评价 Microsoft Edge 浏览器？)，JS内核分别为Jscript(IE8),Chakra(IE9+)
  • Gecko(Firefox使用)，js内核(最新)为OdinMonkey
  • Webkit(Chrome之旧内核,Safari内核),包含WebCore (html?)排版引擎和JavaScriptCore js解析引擎，是苹果从KHTML内核衍生出来的内核，
  • Blink(Chrome和Opera的新内核),Chrome的JavaScript引擎为V8,WebCore排版引擎
  • Presto(Opera旧内核)，不做介绍

####内核工作流程：

  1. 解析HTML构造DOM树：
    同时解析CSS文件及html中内嵌样式，相结合用于用于渲染树的构建
  2. 渲染树构建
    渲染树的节点为一个个包含样式属性的盒模型
  3. 页面布局
    为DOM树节点分配不同坐标
  4. 绘制页面(通过渲染树)

上述过程并非线性同步，而是异步进行，即解析HTML同时进行渲染树构建以及页面布局绘制。这也导致了在这一过程中,JS对DOM的修改会导致重绘

Webkit和Gecko的工作流程有些不同：

获取HTML文档，通过html编译器解析HTML文档为DOM树(html标签转化为DOM节点)，
与此同时，获取到样式表(包括内嵌和外链)并解析为样式规则。DOM树和样式规则
相连接共同作用生成渲染树，渲染树用于布局(这里渲染树和布局的作用是双向的?),浏览器通过生成好的渲染树进行绘制，从而将样式应用到html文档并展示出来。

获取HTML文档，解析html文档为内容槽(用于连接html解析器和文档?)，内容槽转化为内容模型，内容槽同时通过css解析器转化为样式规则，样式规则和内容模型连接生成帧构造(框架树)，帧构造产生框架树(帧 === 框架)，用于重排(reflow)，之后通过绘制从而展现应用样式规则的html文档。

###详细介绍：

####解析：

HTML文档采用DTD格式，非上下文无关语法，所以无法通过常规解析器解析。
解析器解析HTML文档输出DOM树(由DOM元素和属性即节点构成的树结构)

#####解析过程：

  1. 标记化:词法分析，输入内容拆分成多个标记，包括起始结束标记，属性名称和值
  2. 树构建，识别标记同时传递给树构造器，不断识别标记，并构建DOM树

标记化算法：

通过状态机实现，不同状态下遇到不同字符，跳转到不同状态。过于复杂，不再赘述。

树构建算法：

将Document对象作为DOM树根节点，对DOM树不断进行修改。状态机分割的token传递过来后，通过树构建器将其添加到DOM树中，同时，通过堆栈对标记的闭合等问题进行检测。

解析完成后，DOM树构建完毕，页面变为交互状态(加载完成)，之后加载defer的脚本，对DOM树进行修改，完全完毕后，变为”完成”状态。
此时，加载事件将可被触发。

####CSS解析
CSS为上下文无关语法，所以可以通过常用解析器进行解析。

Webkit解析器:
使用Flex和Bison解析生成器，通过CSS语法规则自动创建解析器。
Bison创建自上而下的移位规约解析器。Firefox使用的是自己编写的自上而下的解析器。通过解析器将CSS文件解析为一个个StyleSheet对象，每个对象包含选择器和声明对象(CSS规则)。

####脚本和样式处理顺序

#####脚本:

同步解析，阻塞浏览器进程。如果是外链脚本，下载过程中会阻塞页面解析，等待脚本下载并执行,原因是脚本文件可能修改HTML文档结构，此时并行执行解析和执行脚本任务有可能会产生冲突，所以执行过程必须要阻塞浏览器进程(页面渲染和DOM树生成)。
现代浏览器现在也可以并行下载脚本文件，但是脚本的下载会阻塞其他资源的下载。并行执行问题仍然存在，所以脚本执行仍然会阻塞浏览器进程。
defer脚本会在DOM加载完成，onload事件触发前被执行，但是下载仍在解析到script标签时就开始。
async脚本异步执行脚本(多个脚本异步执行顺序不一定,onload事件前执行)，同样下载是立即下载
动态脚本会立即下载，执行，并且不阻塞浏览器进程。

####样式表:

应用样式表不会修改DOM树，但是脚本有可能请求样式信息。
如果在样式表加载解析过程中，脚本试图获取样式信息就会获得错误信息。
所以不同浏览器采用了不同方案阻止这样的情况发生。
Firefox在样式表加载解析过程中，禁止脚本。而Webkit则会在尝试获取未加载的样式时，禁止该脚本。

####渲染树构建:

DOM树构建同时，渲染树也在进行构建，不同于DOM树的构建，渲染树的节点是根据显示顺序来添加的。

####渲染树和DOM树关系

DOM树中display:none的节点和非可视化节点(head等)不会添加到渲染树中，
DOM树中一个节点也可能对应多个可视化对象

例如，“select”元素有 3 个呈现器：一个用于显示区域，一个用于下拉列表框，还有一个用于按钮。如果由于宽度不够，文本无法在一行中显示而分为多行，那么新的行也会作为新的呈现器而添加。
格式无效的 HTML。例如:根据 CSS 规范，inline 元素只能包含 block 元素或 inline 元素中的一种。如果出现了混合内容，则应创建匿名的 block 呈现器，以包裹 inline 元素。

DOM树中节点位置可能和渲染树中对应节点位置不同。受float和position属性影响,在渲染树中原位上防止占位帧，真正所在位置才放置实际用于渲染的帧。

####Firefox规则树
Firefox中样式对象保存在类似样式上下文树的树结构中

样式上下文包含 端值,

样式上下文分割为多个结构。结构体中包含特定类别的样式信息。
结构中属性均为继承或者均为非继承。可继承属性若在当前节点无定义，则继承自父节点。不可继承属性未定义，则使用默认值。

####渐进式处理

Webkit通过一个标记来表示是否所有顶级样式表均已加载完毕，如果在attach过程中尚未加载完全，则使用占位符并在文档中标注，等加载完毕后重新计算。

####布局

渲染树创建完毕后，并不包含位置和大小信息。通过布局(重排)来计算元素的位置大小。

HTML采用流的布局模型，所以大多数情况下一次遍历即可计算出文档布局大小。
坐标系相对于根框架建立，采用上左坐标。

####Dirty位系统

对某个元素呈现器发生变化做标记，
dirty表示当前元素呈现器变化
children are dirty表示子树中有元素呈现器变化

####全局布局和增量布局

全局布局： 触发了渲染树框架级别的布局变化
增量布局： 只对dirty呈现器进行布局。呈现器为dirty时异步触发增量布局。

####异步布局和同步布局

增量布局： 异步执行。将dirty呈现器的增量布局任务’reflow’命令加入队列，调度程序每次批量执行命令。

请求样式信息(元素布局信息,例clientHeight)可同步触发增量布局。

全局布局：同步触发

####优化

布局仅仅是大小调整或者呈现器的 位置改变而触发，则从缓存中获取呈现器大小，只计算呈现器位置。
有时渲染树中的子树变化时，不影响周围元素的布局。所以无需从根节点重新布局。

####布局处理

布局处理模式：

  1. 父呈现器设置自己的大小
  2. 父呈现器一次处理子呈现器：
    1. 放置子呈现器
    2. 有必要的话，调用子呈现器的布局(例：dirty)，重新计算子呈现器布局
  3. 父呈现器根据子呈现器累加高度(包括内外边距和边框，补白)确定自身高度
  4. dirty位设置为false

####换行

如果呈现器在布局过程中需要换行，则立即停止布局，告知祖先需要换行，父元素创建额外呈现器并进行布局。

####绘制

遍历渲染树，调用呈现器的paint方法并显示在屏幕上。

####全局绘制和增量绘制

部分呈现器变化但不影响整个树，则将其绘制结果(在屏幕上的显示区域)设置为无效，并设置为’dirty区域’。浏览器合并多个’dirty区域’后调用paint。

####绘制顺序

  1. 背景颜色
  2. 背景图片
  3. 边框
  4. 子代
  5. 轮廓

####呈现引擎的线程

呈现引擎为单线程，Firefox和Safari中为浏览器主线程，而Chrome中，因为采用一个多核(一个标签页一个进程)，该线程为当前标签的主线程。

####事件循环

浏览器主线程为时间循环。是永远处于受理状态的一个无限循环，等待时间发生并处理

####CSS2可视化模型

####CSS盒模型

由内容，内边距，边框，外边距组成。
display属性则决定生成的框类型。

####定位

  1. 无定位属性或默认定位属性
  2. 浮动
  3. 相对绝对固定定位

框类型：

block类型：从上到下布局
inline类型：从左到右布局，包含块大小不足以放置下一个inline类型框时，放置到下一行开头，默认(尽量)从包含块顶部开始放置

####浮动：

浮动框会移动到行的左边或者右边。会产生环绕效果。脱离普通流。

####绝对定位和固定定位

脱离普通流。

绝对定位元素的位置相对于最近的定位元素(relative,absolute,fixed均可)
固定定位位置相对于视口。

####分层展示

多个框分散在多个 堆栈上下文中,每个堆栈中，先绘制z轴中后面的元素，然后在顶部绘制前面的元素。重叠时，新绘制元素覆盖之前的元素。
具有z-index属性框形成本地堆栈。视口具有外部堆栈。
即首先看标签顺序，之后看z-index顺序，最后看是否为固定定位。

类数组转化为数组

类数组: arguments 等，

  • 具有length属性，
    • 按照索引方式存储数据
    • 不具有push和pop方法(但是如果对象及其length属性为可读写属性,即可调用数组的push和pop方法)

可以直接在类数组对象上直接调用数组原型的方法:

Array.prototype.push.call(obj,newElement)

需要作为真实数组多次处理时，须转化为数组:

var arr = Array.prototype.slice.call(obj,0);

建议封装到个人常用函数库进行复用.

测试如下:

var obj = {};
obj[0] = '1';
obj[1] = '2';
obj[2] = '3';  
obj.length = 3;  //模拟类数组，即只需要满足开头所说要求，即可视为类数组
alert(arr instanceof Array);  //true
alert(obj instanceof Object);  //true
alert(arr[1]);     //'2'
alert(obj['1']);   //'2'

IE8及以下存在兼容性问题，
需要显式设置类数组的length属性(slice中使用)，否则上述测试中 alert(arr[1]) 返回undefined

attribute和property区别

MDN翻译如下:

attribute拓展了标签，给标签提供了新的行为或者元数据。attribute总是有着一个键值对的表格，赋予attribute标识符以及相关联的值。

####property：

DOM对象的property值通过点取法获取

document.body.innerHTML;

DOM对象作为对象，可以在脚本中自定义property，而自定义property不出现在html文档中
所以，property标识符定义只遵守js标识符命名规范

####attribute:

DOM节点提供以下方法访问attribute(存在兼容问题)

  • hasAttribute
  • getAttribute
  • setAttribute
  • removeAttribute

由于是DOM节点的“属性”，所以遵守HTML标识符规范：

  • 值只能为字符串
  • 不区分大小写
  • html中出现
  • 可以通过DOM对象的属性:attributes对象获取所有attribute组成的一个类数组NamedNodeMap
  • 其实attribute键名也为字符串形式_

JS中操作DOM对象的property只需要点取法或者下标法即可获取，而除了‘共通属性’外的其他attribute，则只能通过JS自带attribute方法获取。html自定义属性(自定义attribute)也能通过attribute一类方法访问

常用attriubte中，id、class、title、style等也作为property附加到DOM对象中了

####特殊‘共通属性’分辨:

class:

HTML中class作为attribute，在getAttribute等方法中，应该传入’class’(字符串形式)
作为property，由于class为保留字，所以改为className

alert(test['class']);  // 尝试通过property方法获取,失败
alert(test['className']); //property中class正确书写为className

style:

alert(test.style);    //返回div1指向的DOM节点的CSSStyleDeclaration对象
alert(test.getAttribute('style'));  //返回一个包裹了对应DOM节点的样式 字符串

PS：attribute方法返回的是行内样式字符串

onclick:

alert(typeof btn.getAttribute('onclick'));  //string
alert(typeof btn.onclick);   //function

个人建议:
当试图将‘属性’看作attribute访问时，最好通过JS自带 attribute 方法进行访问，点取法和下标法用于获取property，易读，
若考虑兼容性，可对特定‘共通属性’作规范.

同名标识符优先级讨论

####局部变量先使用后声明，不影响外部变量

var foo = 1;
function fn(){
    alert(foo);   // undefined 
    var foo = 2;
}
fn();

解析:
局部变量声明提前，即是外部变量有值，仍然按照作用域链顺序，获取到局部变量后返回其值

####形参优先级高于函数名

function foo(foo) {
    alert(foo);   //1
}
foo(1);

解析:
JS解析foo为形参，故输出1，否则输出整个函数

####形参优先级高于arguments

function foo(arguments) {
    alert(arguments); //1
}

JavaScript写类方式

本文讨论面向对象中类的书写方式,暂不考虑类的继承和私有方法属性

####构造函数方法

function Person(name) {
    this.name = name;
    this.getName = function(){
        return this.name;
}

根据参数不同，构造拥有不同属性的对象实例
对象实例的方法每次都会被创建,浪费资源，例如getName这种方法完全可以所有同类型实例对象共享

改进，外部函数作为共享方法，所有对象的对应共享方法引用指向对象属性,通过对象属性实现对共享方法的调用

function getName() {
    return this.name;
}
function Person(name) {
    this.name = name;
    this.getName = getName;
}

####原型方式

所有对象实例的属性均为共用，无法通过参数对对象实例属性进行定制

function Person() {}  //作为构造函数调用时，返回一个默认对象,包含Object原型对象的所有属性
Person.prototype.name = 'TalkLee';
Person.prototype.getName = function(){
    return this.name;
};

构造函数+原型
原型：用于生成共享方法
构造函数:用于定制拥有不同对象的对象实例

function Person(name) {
    this.name = name;
}
Person.prototype.getName = function(){
    return this.name;
};

结构上共享方法和对象实例属性关系不够清晰
将共享方法放置在构造函数中定义，清晰分辨出共享方法的性质

function Person(name) {
    this.name = name;
    Person.prototype.getName = function() {
        return this.name;
    }
}

但每次生成对象实例时都需要生成匿名函数，并且传递引用

故加入判断

function Person(name) {
    this.name = name;
    if(Person.hasInit == undefined ) {   //第一次生成对象实例
        Person.prototype.getName = function(){
            return this.name;
        }
    }
    Person.hasInit = 1;  //定义以后给对象标志属性赋值,以后即可用此对象进行判断
}

function $class(constructor,prototype) {   //生成类函数

具名函数的多种调用方式

背景代码:

function foo(){
    alert('a');
}

方法1:

括号()运算符调用

foo();

方法2&3:

call和apply方法

foo.call();  //call和apply中传入的第一个参数为函数调用中作用域(this对象),
单纯为调用函数的话，传入变量常量任何均可
foo.apply();

方法4：

new构造函数调用方法

new foo()

注意以下情况:

    function foo(param){
    alert(param);
    return param;   //有返回值
}

var a = new foo({ b : '1'});  //返回传入对象
var a = new foo('OK');    //返回另一个对象

若函数声明中返回对象，则构造函数调用时返回指定的对象
若函数声明中返回基本类型(包括null和undefined),构造函数调用时实际返回函数语句中生成的对象(通过this给生成的对象添加属性)

function foo(param){
    alert(param);
    this.b = '1';
    return null;
}

上例中return语句无效,返回对象，包含属性b.

若函数声明中声明新对象，并且通过this给构造函数生成对象添加属性，又会出现什么情况呢?

若无返回值或者返回基本类型或者返回this，则返回this产生的对象，
若返回构造函数中声明的对象，则返回指定的对象(工厂模式的写法)

function foo(param){
    this.pro = 'this';
    var obj = {};
    obj.name = 'foo';
    obj.msg = param;
    //return this;  返回对象中包括pro属性
    //return obj;  返回对象为obj
}

工厂模式:

function foo(param){
    var obj = {};
    obj.msg = param;
    return obj;
}
var a = new foo('2');  //构造函数方式调用
var b = foo('3');  //函数调用,两种方法返回结果相同，原因上面已经讨论过了
